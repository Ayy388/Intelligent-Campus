## 1. 项目概述

### 1.1 项目背景

智能校园管理系统是一个面向高校的一站式校园管理与服务平台，旨在通过信息化手段提升校园管理效率，改善师生在校体验。当前高校普遍面临以下痛点：

- **信息分散**：课程、成绩、通知、社团等信息分布在不同的系统或纸质渠道中
- **流程繁琐**：请假、活动审批等行政流程依赖线下跑腿，效率低下
- **沟通低效**：师生之间、社团成员之间缺乏便捷的沟通渠道
- **管理粗放**：排课、选课、成绩统计等教务工作依赖人工操作，易出错

本系统通过统一平台整合上述需求，覆盖**教务管理、行政服务、社团活动、校园生活、信息沟通**等多个领域，为不同角色用户提供差异化的功能入口。

### 1.2 项目目标

- **统一身份认证**：构建 JWT 单点登录平台，实现基于角色的权限管理
- **教务管理数字化**：课程管理 → 排课 → 选课 → 成绩录入 → 统计分析，全链路覆盖
- **行政事务在线化**：通知发布、请假申请与审批、场地预约等流程全部线上完成
- **校园生活平台化**：社团管理、活动组织、失物招领等服务于一体
- **沟通渠道便捷化**：站内私信、公告推送、AI 智能助手多渠道触达

### 1.3 目标用户

| 用户角色 | 人数规模 | 核心诉求 |
|---------|---------|---------|
| 在校学生 | $n$ × 1000 | 选课查成绩、看课表、参与社团与活动、处理请假等事务 |
| 授课教师 | $n$ × 50 | 课程教学管理、成绩录入、查看课表 |
| 辅导员 | $n$ × 10 | 管理 1 个专业+年级下的 2-3 个班级，学生事务审批 |
| 系统管理员 | 1-2 | 系统配置、基础数据维护、业务审核与监管 |

## 2. 用户角色定义

### 2.1 角色总览

| 角色编码 | 角色名称 | 角色 ID | 权限层级 | 默认首页 |
|---------|---------|---------|---------|---------|
| student | 学生 | 1 | 最低：仅操作个人数据 | 仪表盘 |
| teacher | 教师 | 2 | 中等：管理所授课程及相关数据 | 仪表盘 |
| admin | 管理员 | 3 | 最高：系统级配置与所有业务审核 | 仪表盘 |
| counselor | 辅导员 | 4 | 较高：学生事务管理 | 仪表盘 |

### 2.2 学生（Student）

**身份描述：** 在校学生是系统最主要的用户群体，使用系统进行学习生活相关的各项操作。学生以个人为中心使用系统，不涉及管理他人数据。

**核心使用场景：**

```
作为学生，我能够……
├─ 账号与登录
│  ├─ 使用学号和密码登录系统
│  ├─ 修改个人资料（头像、邮箱、手机号）
│  └─ 修改登录密码
├─ 教务相关
│  ├─ 浏览可选课程列表，查看课程详情（教师、时间、容量等）
│  ├─ 自主选课（选修课）和退选
│  ├─ 查看我已选的课程列表及选课状态
│  ├─ 按周查看个人课程表（颜色区分，自动根据当前周高亮）
│  └─ 查询我的成绩，含 GPA 计算、成绩分布柱状图、学期趋势图
├─ 行政事务
│  ├─ 提交请假申请（病假/事假/公假，可附图片）
│  ├─ 查看我的请假记录和审批状态
│  └─ 查看学校通知公告（分类筛选、标记已读）
├─ 社团与活动
│  ├─ 浏览所有社团列表及详情
│  ├─ 创建新社团（待管理员审批）
│  ├─ 申请加入社团 / 退出社团
│  ├─ 查看我的社团空间（成员、活动）
│  ├─ 浏览校园活动并报名/取消报名
│  └─ 查看我创建和报名的活动
├─ 校园生活
│  ├─ 校园卡虚拟充值（预设金额快捷选择）
│  ├─ 查看充值记录
│  ├─ 发布失物/招领信息（可带图片）
│  └─ 标记失物招领为已解决
├─ 沟通
│  ├─ 给其他用户发送私信
│  ├─ 查看私信会话和聊天历史
│  └─ 使用 AI 助手进行校园问答
└─ 个人工具
   ├─ 管理个人待办事项（优先级/截止日期/完成状态）
   └─ 仪表盘查看个人数据概览
```

### 2.3 教师（Teacher）

**身份描述：** 授课教师负责课程教学管理相关操作。教师管理自己教授的课程，不涉及系统级配置。

**核心使用场景：**

```
作为教师，我能够……
├─ 教务教学
│  ├─ 查看我教授的所有课程列表（按学期筛选）
│  ├─ 查看某门课程的学生名单及详细信息
│  ├─ 录入学生成绩（数值制0-100/等级制优-差）
│  ├─ 修改已录入的成绩
│  ├─ 查看各课程成绩录入进度（已录/未录人数）
│  └─ 查看课程成绩统计（平均分/最高分/及格率/分布）
├─ 排课（教师可查看，排课操作权限视学校策略而定）
│  ├─ 查看全校课表（按班级/教室/教师维度）
│  └─ 查看我的课程在课表中的位置
├─ 通知
│  ├─ 发布通知公告
│  └─ 查看已发布的通知列表
├─ 活动
│  ├─ 创建校园活动
│  ├─ 查看我发布的活动列表
│  └─ 对已报名的活动进行确认和总结
├─ 沟通
│  ├─ 发送/接收私信
│  ├─ 发布公告推送（向特定角色群发）
│  └─ 使用 AI 助手
└─ 个人工具
   ├─ 仪表盘查看教学数据概览（课程数/学生数/待办等）
   └─ 管理个人待办事项
```

### 2.4 辅导员（Counselor）

**身份描述：** 辅导员负责特定年级专业下 2-3 个班级的学生日常管理与关怀，侧重学生事务审批和班级管理。

> **管理范围规则：** 每位辅导员被分配到 **1 个专业 + 1 个年级** 的组合下，管理该组合中的 **2-3 个班级**。例如：计算机科学与技术专业 2025 级下，计科 2501 班和计科 2502 班由同一位辅导员管理。

**核心使用场景：**

```
作为辅导员，我能够……
├─ 学生事务
│  ├─ 管理所辖班级（2-3 个班，同一专业+年级下）的基础信息
│  ├─ 审批所辖班级学生的请假申请（通过/驳回并填写原因）
│  ├─ 查看所辖学生的请假记录
│  └─ 查看所辖班级的数据概览
├─ 通知
│  ├─ 发布通知公告
│  ├─ 发布公告推送（向所辖班级群发）
│  └─ 查看已发布的通知
├─ 沟通
│  ├─ 发送/接收私信
│  └─ 使用 AI 助手
└─ 个人工具
   ├─ 仪表盘查看管辖班级数据概览
   └─ 管理个人待办事项
```

### 2.5 系统管理员（Admin）

**身份描述：** 管理员是系统的最高权限角色，负责系统配置与全局管理。管理员可以操作所有模块的所有功能。

**核心使用场景：**

```
作为管理员，我能够……
├─ 系统管理
│  ├─ 用户管理：查看/新增/编辑/删除/启禁用所有用户
│  ├─ 部门管理：部门 CRUD
│  ├─ 专业管理：按部门维护专业 CRUD
│  ├─ 年级管理：年级 CRUD
│  └─ 班级管理：按部门→专业→年级级联维护班级 CRUD
├─ 教务管理
│  ├─ 课程管理：创建/编辑/删除/查询所有课程
│  ├─ 排课管理：以课表视图为课程分配时间和教室，检测冲突
│  ├─ 选课管理：分配教学班级、确认/取消开课、查看选课学生
│  ├─ 学期管理：学期 CRUD
│  ├─ 培养方案管理：按专业+年级维护培养方案及课程项
│  ├─ 培养方案自动开课：一键生成学期课程
│  └─ CSV 批量导入课程
├─ 业务审核
│  ├─ 社团审核：审批社团创建/解散/成员申请
│  ├─ 活动审核：审批校园活动发布
│  └─ 场地预约审核：审批场地使用申请
├─ 数据统计
│  ├─ 成绩统计：全课程成绩分布、平均分、及格率
│  └─ 场地利用率统计（可选）
├─ 通知与沟通
│  ├─ 发布通知公告和公告推送
│  ├─ 发送/接收私信
│  └─ 使用 AI 助手
└─ 个人工具
   ├─ 仪表盘查看全局数据概览
   └─ 管理个人待办事项
```

## 3. 功能需求

### 3.1 系统公共模块

| 编号 | 功能 | 描述 | 输入 | 输出/结果 | 学生 | 教师 | 辅导员 | 管理员 | 优先级 |
|------|------|------|------|-----------|:----:|:----:|:------:|:------:|:------:|
| SYS-01 | 用户登录 | 用户名/密码登录，服务端校验密码并返回 JWT Token | 用户名、密码 | Token + 用户基本信息 | ✓ | ✓ | ✓ | ✓ | P0 |
| SYS-02 | 获取当前用户信息 | 根据 Token 获取当前登录用户的详细信息（含角色、部门等） | 无（从 Token 解析） | 用户完整信息 | ✓ | ✓ | ✓ | ✓ | P0 |
| SYS-03 | 修改个人资料 | 修改姓名、邮箱、头像、手机号等信息 | 待修改的字段 | 更新成功/失败 | ✓ | ✓ | ✓ | ✓ | P1 |
| SYS-04 | 修改密码 | 验证旧密码后更新为新密码 | 旧密码、新密码 | 更新成功/失败 | ✓ | ✓ | ✓ | ✓ | P0 |
| SYS-05 | 上传头像 | 上传用户头像图片（限制 2MB，jpg/png） | 图片文件 | 头像 URL | ✓ | ✓ | ✓ | ✓ | P1 |
| SYS-06 | 上传图片 | 通用图片上传（请假附件、失物招领等，限制 5MB） | 图片文件 | 图片 URL | ✓ | ✓ | ✓ | ✓ | P1 |

**边界条件与约束：**
- 登录失败连续 5 次不锁定（当前版本不实现）
- Token 有效期 2 小时，过期需重新登录
- 密码需满足长度 ≥ 6 位

### 3.2 用户管理模块（sys）

| 编号 | 功能 | 描述 | 前置条件 | 权限 | 优先级 |
|------|------|------|---------|------|:------:|
| USR-01 | 用户列表 | 分页查询所有用户，支持按角色/姓名/学号筛选 | 已登录 | admin | P0 |
| USR-02 | 新增用户 | 创建用户（设置用户名/密码/角色/部门/班级等），密码默认 123456 | - | admin | P0 |
| USR-03 | 修改用户 | 编辑用户信息（角色/部门/班级等均可修改） | 用户存在 | admin | P0 |
| USR-04 | 删除用户 | 删除指定用户（系统保留操作记录） | 用户存在，非当前登录用户 | admin | P1 |
| USR-05 | 启禁用用户 | 启用/禁用用户账号，禁用后无法登录 | 用户存在，非当前登录用户 | admin | P0 |
| USR-06 | 部门管理 CRUD | 部门增删改查（删除时检查是否有引用） | - | admin | P0 |
| USR-07 | 专业管理 CRUD | 专业增删改查，创建时选择所属部门，列表支持按部门过滤 | 部门存在 | admin | P0 |
| USR-08 | 年级管理 CRUD | 年级增删改查 | - | admin | P0 |
| USR-09 | 班级管理 CRUD | 班级增删改查，级联选择：部门 → 专业 → 年级，同时指定辅导员和班主任姓名 | 部门/专业/年级存在 | admin/counselor | P0 |

**业务规则：**
- 删除部门/专业/年级时需检查是否被引用，被引用则禁止删除并提示
- 用户名唯一，不可重复
- 学生必须关联班级，教师/辅导员/管理员不必关联
- **辅导员管理范围规则：** 一个辅导员只能被分配到**同一专业 + 同一年级**下的班级。当为班级选择辅导员时，系统校验该辅导员已有的班级是否属于同一专业和年级；若超出范围则提示"该辅导员已有其他专业/年级的班级，请选择本专业/年级的辅导员"。一个辅导员管理的班级数量建议不超过 3 个。

### 3.3 教务管理模块 — 课程管理（edu）

#### 3.3.1 课程基础管理

| 编号 | 功能 | 描述 | 前置条件 | 权限 | 优先级 |
|------|------|------|---------|------|:------:|
| EDU-01 | 课程列表查询 | 分页查询课程，支持按名称/课程代码/状态/教师/学期组合筛选 | 已登录 | authenticated | P0 |
| EDU-02 | 新增课程 | 创建课程，填写课程名/代码/学分/学时/教师/课程类型（必修/选修）/课程描述 | 学期已创建 | admin/teacher | P0 |
| EDU-03 | 修改课程 | 编辑课程基本信息 | 课程存在，未被选课引用时部分字段可改 | admin/teacher | P0 |
| EDU-04 | 删除课程/取消开课 | 删除未发布的课程；已发布课程只能取消开课（状态 → 已取消），不可物理删除 | 课程存在 | admin | P1 |
| EDU-05 | CSV 导入课程 | 通过 CSV 文件批量导入课程（模板：课程名/代码/学分/学时/教师/类型） | 学期已创建 | admin | P1 |

**业务规则：**
- 课程代码 (`courseCode`) 在学期内唯一
- 删除课程时需检查是否存在选课记录约束（后面有专门的约束说明）
- 课程创建时默认状态为「未发布(0)」

#### 3.3.2 排课管理

| 编号 | 功能 | 描述 | 前置条件 | 权限 | 优先级 |
|------|------|------|---------|------|:------:|
| EDU-06 | 排课列表 | 以列表形式展示所有已排/未排课程，标注排课状态（已排/未排/部分排） | 课程已创建 | admin/teacher | P0 |
| EDU-07 | 新增排课 | 为课程指定上课时间与地点：选择星期（周一至周五）、节次（第1-4大节）、周模式（全周/单周/双周）、起止周、教室 | 课程已创建、教室已录入 | admin | P0 |
| EDU-08 | 排课冲突检测 | 提交排课时自动校验三项冲突：①同一教室同一时段是否已被占用 ②同一位教师同一时段是否已有其他课程 ③同一班级同一时段是否已有其他课程 | 有排课提交 | admin | P0 |
| EDU-09 | 修改排课 | 调整已有排课记录的时间或教室，修改后重新触发冲突检测 | 排课记录存在 | admin | P0 |
| EDU-10 | 删除排课 | 删除某条排课记录 | 排课记录存在，课程未进入选课阶段 | admin | P0 |
| EDU-11 | 课表多维查看 | 按三种维度查看课表：①按班级查看 ②按教师查看 ③按教室查看 | 已有排课数据 | authenticated | P1 |
| EDU-12 | 个人课表查看 | 按周查看当前登录用户的课程表，颜色区分课程，自动定位当前周 | 已登录 | authenticated | P0 |
| EDU-13 | 课表导出 | 将个人课表导出为文本文件 | - | authenticated | P1 |

**排课冲突检测细则：**

```
冲突类型              检测条件
─────────────────────────────────────────────────────
教室冲突             相同教室 + 相同星期 + 相同节次 + 周模式有交集 → 冲突
教师时间冲突          相同教师 + 相同星期 + 相同节次 + 周模式有交集 → 冲突
班级时间冲突          相同班级 + 相同星期 + 相同节次 + 周模式有交集 → 冲突
```

**周模式交集规则：**
- `全周 + 全周` → 有交集
- `全周 + 单周` → 有交集（全周覆盖单周）
- `全周 + 双周` → 有交集
- `单周 + 单周` → 有交集
- `双周 + 双周` → 有交集
- `单周 + 双周` → 无交集

#### 3.3.3 选课管理

| 编号 | 功能 | 描述 | 前置条件 | 权限 | 优先级 |
|------|------|------|---------|------|:------:|

**选课约束条件：**

```
选课成功条件（全部满足）：
  ✓ 课程状态 == 选课中(1)
  ✓ 当前时间在选课期内
  ✓ 课程容量未满（enrolled < capacity）
  ✓ 学生与该课程无重复选课记录
  ✓ 学生所在的班级已分配该课程（选修课不受此限）

选课失败原因：

  ✗ 已选过该课程（不可重复选）
  ✗ 该课程未向学生所在班级开放（限必修课）

  ✗ 系统自动分配的必修课不可退选
  ✗ 课程已确认开课后不可退选
```

## 4. 业务流程描述

### 4.1 教务全流程总览

```
学期管理 ─→ 课程创建 ─→ 分配班级 ─→ 排课 ─→ 发布选课 ─→ 学生选课
    │                                                      │
    └──────────────── 学期结束 ←──── 成绩录入 ←──── 确认开课 ──┘
```

**各阶段角色：**

| 阶段 | 执行者 | 前置条件 |
|------|--------|---------|
| 创建学期 | 管理员 | 无 |
| 创建课程 | 管理员/教师 | 学期已创建 |
| 分配班级 | 管理员 | 课程已创建 |
| 排课（分配时间/教室） | 管理员 | 课程已创建，班级已分配 |
| 发布选课 | 管理员 | 课程信息完整 |
| 选课 | 学生 | 课程状态=选课中 |
| 确认开课 | 管理员 | 选课截止 |
| 成绩录入 | 教师 | 课程已确认 |
| 学期结束 | 系统/管理员 | 学期设置中有结束日期 |

### 4.2 课程创建 → 排课 → 选课 → 开课流程

```
参与者：管理员、教师、学生、系统
触发条件：新学期开始前

─── 准备阶段 ───
1. 管理员创建学期（设置学年/名称/起止日期/周次）
2. 管理员/教师创建课程（课程名/代码/学分/学时/教师/类型）
3. 管理员为课程分配教学班级（指定哪些班级上这门课）
4. 管理员进行排课 —— 为课程分配：
   a. 上课星期（周一至周五）
   b. 上课节次（第1、2、3、4大节）
   c. 周模式（全周/单周/双周）
   d. 起止周
   e. 教室
5. 系统在每次排课时自动检测三项冲突：
   5a. 教室冲突：同一教室同一时段已有排课 → 提示"该教室已被占用"
   5b. 教师冲突：该教师同一时段已有排课 → 提示"该教师时间冲突"
   5c. 班级冲突：该班级同一时段已有排课 → 提示"该班级时间冲突"
6. 管理员可查看完整课表（按班级/教师/教室维度），确认无误

─── 选课阶段 ───
7. 管理员将已排课的课程状态改为「选课中」
8. 系统自动为必修课（is_required=1）的所有学生生成选课记录
9. 学生在选课期间浏览可选课程列表，可进行自主选课
   9a. 系统校验：课程容量未满 & 不重复选课 → 成功，创建选课记录
   9b. 若容量已满 → 提示"选课人数已满"
10. 学生可在选课期间退选选修课

─── 开课阶段 ───
11. 选课截止后，管理员确认课程开课：
    11a. 选课人数达标（≥ minStudents）→ 确认开课（状态→已确认）
    11b. 选课人数不足 → 可选择取消开课（状态→已取消）
12. 课程状态变为「已确认」后，教师可查看学生名单
```

**状态流转：**

```
未发布(0) ─→ 选课中(1) ─→ 已确认(2) [正常开课]
                         └→ 已取消(3) [人数不足或管理员取消]
              ↑
              └ 取消开课的课程不可再恢复
```

### 4.3 请假审批流程

```
参与者：学生、辅导员/管理员
触发条件：学生需要请假

1. 学生进入请假申请页面
2. 填写请假信息：
   a. 选择请假类型（病假/事假/公假）
   b. 选择开始时间和结束时间
   c. 填写请假原因（必填，不少于10字符）
   d. 可选上传附件（图片格式，最大5MB）
3. 系统校验：
   a. 起止时间不能早于当前时间
   b. 开始时间不能晚于结束时间
   c. 同一时段无其他待审批请假
4. 校验通过 → 创建请假记录，状态=待审批
5. 辅导员/管理员登录后看到待审批请假列表
6. 审批人点击查看请假详情（含附件预览）
7. 审批人操作：
   7a. 通过 → 状态变为「已通过」
   7b. 驳回 → 填写驳回原因 → 状态变为「已驳回」
8. 学生可查看审批结果和驳回原因（如有）
```

**状态流转：**

```
待审批(0) ───→ 已通过(1)
          └──→ 已驳回(2)
```

### 4.4 排课冲突检测流程

```
参与者：管理员
触发条件：管理员提交排课或修改排课

1. 管理员选择课程，输入排课信息（星期/节次/周模式/教室）
2. 管理员提交排课
3. 系统执行三项冲突检测：

   ┌─ 检测1：教室冲突
   │  查询条件：相同教室 + 相同星期 + 相同节次 + 周模式有交集
   │  如果冲突 → 提示"该教室在X节已被占用"
   │
   ├─ 检测2：教师时间冲突
   │  查询条件：相同教师(课程) + 相同星期 + 相同节次 + 周模式有交集
   │  如果冲突 → 提示"该教师在X节已有课程"
   │
   └─ 检测3：班级时间冲突
      查询条件：相同班级(课程班级) + 相同星期 + 相同节次 + 周模式有交集
      如果冲突 → 提示"该班级在X节已有课程"

4. 若有任意一项冲突 → 排课失败，返回冲突详情
5. 若无冲突 → 排课成功，更新课程排课信息
```

### 4.5 成绩录入流程

```
参与者：教师
触发条件：课程教学结束

1. 教师进入「成绩录入」页面
2. 选择需要录入成绩的课程（仅显示状态=已确认且该教师教授的课程）
3. 系统展示该课程选课学生列表（含学号/姓名/班级）
4. 教师输入成绩：
   4a. 数值制：输入 0-100 的整数
   4b. 等级制：从下拉选择「优秀/良好/中等/及格/不及格」
5. 教师可逐个学生提交，也可填写多人后批量提交
6. 提交时校验：
   a. 数值制 0-100 范围内
   b. 一个学生一条成绩记录（不可重复提交）
7. 提交成功 → 成绩保存，学生端可见
8. 教师可修改/删除已录入成绩
9. 教师可在「成绩提交状态」页查看各课程的录入进度
```

### 4.6 社团创建与成员管理流程

```
参与者：学生（发起人/申请人）、社长、管理员
触发条件：学生想要创建/加入社团

─── 创建社团 ───
1. 学生填写社团创建申请表（名称、描述，可选 Logo）
2. 系统创建社团记录（状态=待审批），创建人自动成为待定社长
3. 管理员在社团审批页面查看待审批列表
4. 管理员操作：
   4a. 审批通过 → 状态变为已激活，创建人正式成为社长
   4b. 审批驳回 → 社团被拒绝，创建人可查看驳回原因

─── 加入社团 ───
5. 用户浏览已激活的社团列表
6. 用户点击「申请加入」并填写申请理由
7. 社长（或管理员）查看待审批的成员申请
8. 审批操作：
   8a. 通过 → 用户成为社团成员（角色=member）
   8b. 拒绝 → 申请被退回

─── 社团运营 ───
9. 社长可管理成员：设置副社长、移除成员
10. 社长可发布社团内部活动
11. 成员可申请退出社团（非社长）
12. 社长可发起解散申请 → 管理员确认后解散
```

**社团状态流转：**

```
创建 → 待审批 ─→ 已激活 ─→ 申请解散 ─→ 已解散
               └→ 已驳回          └→ 直接解散
```

### 4.7 活动发布与审批流程

```
参与者：任何用户（发起人）、管理员、其他用户（报名人）
触发条件：用户想要举办校园活动

─── 发布阶段 ───
1. 用户填写活动信息（标题/描述/地点/起止时间/人数上限/类别/封面图）
2. 提交后状态=待审批，等待管理员审核
3. 发起人可在审批前撤回申请

─── 审批阶段 ───
4. 管理员查看待审批活动列表
5. 管理员操作：
   5a. 通过 → 活动状态变为已审批
   5b. 驳回 → 填写驳回原因，状态变为已驳回

─── 进行阶段 ───
6. 发起人确认活动开始（状态→已确认/进行中）
7. 其他用户在活动广场浏览已审批的活动
8. 用户报名参加活动（需人数未满）
9. 用户可在活动开始前取消报名

─── 收尾阶段 ───
10. 活动结束后，发起人提交活动总结
11. 发起人结束活动（状态→已结束）
```

**活动状态流转：**

```
待审批(0) ───→ 已审批(1) ───→ 已确认(3) ───→ 已结束(4)
          │          （审批通过）    （确认开始）    （提交总结后完结）
          └──→ 已驳回(2)
          │
          └──→ [撤回]（发起人自行取消）
```

### 5.1 性能需求

| 指标 | 要求 | 说明 | 测量方式 |
|------|------|------|---------|

| 数据库连接池 | 20-50 | HikariCP 连接池配置 | 配置参数 |

### 5.2 安全需求

| 类别 | 要求 | 实现方式 |
|------|------|---------|
| 身份认证 | 所有 API（除登录外）必须携带有效 JWT Token | JwtAuthenticationFilter 拦截器 |
| 角色鉴权 | API 级别 + 前端菜单双重控制 | SecurityConfig 路径规则 + @PreAuthorize + 前端 v-if |
| 密码安全 | BCrypt 加密存储，不可逆 | Spring Security BCryptPasswordEncoder |
| 防重放 | JWT 有效期 2 小时，过期失效 | Token 解析时校验 exp |
| 文件上传安全 | 限制类型（仅图片）、大小（头像 2MB、其他 5MB） | MultipartFile 校验 |
| 跨域防护 | 允许从 dev 域名发起的请求 | CorsConfig 配置允许的来源 |
| XSS 防护 | 前端展示内容转义 | Vue 模板默认转义 |
| 配置安全 | JWT 密钥、API 密钥从环境变量读取，不硬编码 | application.yml ${} 占位符 |

### 5.3 可用性需求

| 需求 | 说明 |
|------|------|
| 加载状态 | 所有列表/详情页面在数据返回前显示 loading 动画（Element Plus v-loading） |
| 操作反馈 | 所有增删改操作完成后必须有成功/失败的 Toast 提示（ElMessage） |
| 空状态 | 列表无数据时显示「暂无数据」插图提示，非空白页 |
| 错误兜底 | 接口请求失败时展示友好的错误提示，不出现白屏或系统异常堆栈 |
| 表单验证 | 所有表单提交前进行前端校验（必填/格式/长度），后端二次校验 |
| 确认弹窗 | 所有删除/退选/取消等不可逆操作需弹出确认对话框 |
| 404 页面 | 不存在的路由自动跳转仪表盘 |

### 5.4 兼容性需求

| 要求 | 说明 |
|------|------|
| 浏览器 | Chrome 90+ / Firefox 90+ / Edge 90+ （最近 2 个主要版本） |
| 分辨率 | 适配 1280px 以上桌面端，不做移动端适配 |
| Node.js | >= 18.x |
| Java | 17+ |

### 5.5 数据一致性需求

| 场景 | 要求 | 措施 |
|------|------|------|
| 选课并发更新 | 选课时 enrolled 计数不超 capacity | CAS 更新（UPDATE ... WHERE enrolled < capacity） |

| 并发选课 | 同一时刻多人选同一门课不会超卖 | 乐观锁/数据库行锁 |
