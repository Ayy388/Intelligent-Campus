## 2. 阶段总览

### 2.1 整体路线图

```
Phase A: 基础设施（CI + 配置安全）
Phase B: 质量建设（后端测试 + 前端测试 + Swagger）
Phase C: 技术债务（类型系统 + 错误处理 + 事务）
Phase D: 排课功能实现（新功能）
Phase E: 架构增强（Redis + Docker）
Phase F: 交付（README + 清理）
```

### 2.2 工作量估算（Phase A-C 实际执行修正版）

| 阶段 | 模块 | 预估文件数 | 预估对话数 | 实际对话数 | 差异分析 |
|:----:|------|:---------:|:----------:|:----------:|---------|
| A | CI 配置 + 安全加固 | 4-6 文件 | 1 | 1 | ✅ 符合预期 |
| B1 | 后端测试 | 10 文件 | 1 | 1 | ✅ 符合预期（3 agent 并行有效）|
| B2 + B3 | 前端测试 + Swagger | 8 文件 | 1 | 1 | ⚠️ 实际由 agent 和手工分别完成 |
| C1 | 前端类型系统 | 50 文件 | 1 | 1 | ❌ 1 个 agent 改 50 文件导致 30+ 编译错误，修复时间超预期 |
| C2 | 错误处理修复 | 2-3 文件 | 0.5 | 0.5 | ✅ 后端改动小，前端空 catch 实际不存在 |
| C3 | 事务 + 配置安全 | 3 文件 | 0.5 | 0.5 | ✅ |
| D | 排课功能 | 8-12 文件 | 2-3 | — | 🔜 |
| E | Redis + Docker | 6-8 文件 | 2 | — | 🔜 |
| F | README + 清理 | 5-8 文件 | 1 | — | 🔜 |

**关键教训：** C1 的「预估 15-20 文件」严重低估。10 个 types 文件 + 12 个 api 文件 + 25 个 View = 50 文件。**下次预估要乘 2-3 倍。**

### 2.3 Agent 执行效率对比

| 执行方式 | Phase | 文件数 | agent 耗时 | 后修复耗时 | 总耗时 | 评价 |
|---------|:-----:|:------:|:----------:|:----------:|:------:|------|
| **3 agent 并行** | B（测试） | 10 后端 + 5 前端 + Swagger | ~8 min × 3 | 6 min（集成测试修复）| 14 min | 👍 高效，并行减少了等待 |
| **单 agent 全包** | C1（类型系统） | 50 文件 | ~12 min | 15 min（30+ 编译错误修复）| 27 min | 👎 改得多错得多，修复时间反超 |
| **手工 + 1 agent** | C2-C3 | 5 文件 | ~3 min | 0 | 3 min | 👍 小任务自己做更快 |

**结论：** 不是 agent 越多越好，是 agent 粒度越细越好。一个 agent 的最佳改动量是 5-10 文件。

### Phase C：技术债务修复（预估：3-4 次对话）

**目标：** 代码质量达到作品级标准

#### C-01：前端类型系统

**实施顺序：** `types/` 基类 → API 层 → Store → View

```
步骤 1：创建 src/types/ 目录
  ├── api.ts            → Result<T>, PageResult<T>, 请求参数类型
  ├── user.ts           → User, LoginRequest, LoginResponse
  ├── course.ts         → Course, CourseSelection, Grade
  ├── admin.ts          → Notification, LeaveApplication
  ├── club.ts           → Club, ClubMember, Venue
  ├── activity.ts       → Activity, ActivityRegistration
  ├── life.ts           → CardRecharge, LostFound
  ├── message.ts        → Conversation, Message, Announcement
  ├── todo.ts           → Todo
  └── edu.ts            → Semester, TrainingPlan, Schedule

步骤 2：api/*.ts — 替换 any 为具体类型（10 个文件）

步骤 3：store/*.ts — 替换 any（user.ts, app.ts）

步骤 4：关键 View 替换 any
  优先：edu/ (CourseSelection, GradeQuery, ScheduleView)
  其次：admin/ (LeaveApply, LeaveApproval)
  最后：其余 View
```

**风险：** 类型定义需要与后端 Entity 字段一一对应，容易遗漏新字段。前端 50+ View 文件不可能一次全部改完。采用「核心页面优先，其余逐步推进」策略。

#### C-02：错误处理修复

**实施顺序：** 后端 → 前端 → 统一模式


#### D-01：冲突检测 Service（Agent 1 — 约 5-10 min）

**文件边界：** 只新建 `backend/src/main/java/com/campus/module/edu/service/ScheduleConflictDetector.java`

**不需要改任何现有文件。**

```java
public class ScheduleConflictDetector {
    // 返回值：无冲突返回空 List，有冲突返回冲突描述列表
    static List<String> detect(
        List<Map<String, Object>> newSchedule,   // 新排课数据（JSON 数组解析后）
        Long courseId,
        Long teacherId,
        List<Long> classIds,                     // 该课程关联的班级 ID 列表
        CourseService courseService,              // 用于查询已存在的排课
        CourseClassMapper courseClassMapper
    ) {
        // 1. 教室冲突：同一时间+同一教室
        // 2. 教师冲突：同一时间+同一教师
        // 3. 班级冲突：同一时间+同一班级（通过 edu_course_class 查询）
        // 4. 周模式交集规则：all+all→冲突, odd+even→不冲突, all+odd→冲突
    }
}
```

**冲突检测规则（必须穷举测试覆盖）：**

| 现有周模式 | 新周模式 | 是否冲突 |
|-----------|---------|:--------:|
| all | all | ✅ 冲突 |
| all | odd | ✅ 冲突 |
| all | even | ✅ 冲突 |
| odd | odd | ✅ 冲突 |
| odd | even | ❌ 不冲突 |
| even | even | ✅ 冲突 |
| even | odd | ❌ 不冲突 |

**验证标准：**
```
mvn test -Dspring.profiles.active=test -Dtest=ScheduleConflictDetectorTest
→ 覆盖 12+ 测试用例（3 种冲突类型 × 4 种边界条件）
```

## 4. 风险与应对

### 4.1 核心理念

> 风险不是用来「消除」的，是用来「管理」的。每个风险确认发生后，执行对应的应急预案（Plan B），而不是停下来讨论。

### 4.2 风险分级定义

### 4.10 文档校验 — 开发过程中保持与规格一致

**每次实施代码改动前，必须执行以下三步校验：**

```
步骤 A：定位文档
  这次改动涉及哪个模块？
  ├─ 打开 R0 找到对应功能定义
  ├─ 打开 R1c 找到对应接口定义
  ├─ 打开 R1b 确认涉及的数据库表结构
  └─ 打开 R1a 确认模块依赖和架构约束

步骤 B：对照三问
  Q1: R0 中有这个功能吗？权限定义是否匹配？
  Q2: R1c 的接口定义和我要实现的一致吗？（路径/方法/参数/响应体）
  Q3: R1b 的表结构支持我的数据需求吗？（字段名/类型/约束）

步骤 C：偏差处理
  ┌─ 实现与文档一致 → 正常开发
  ├─ 实现需要偏离文档 →
  │    先更新文档再开发（文档优先原则）
  │    改动后通知你 Review 文档变更
  └─ 文档中有但实际不需要 →
       先删除/修改文档再跳过
```

**三不原则：**
```
文档没写的功能          → 不擅自加
文档定义了的接口         → 不擅自改（格式/路径/权限）
文档中的设计决策         → 不擅自推翻
```

| 粒度 | 改动范围 | 结果 |
|------|---------|------|
| ✅ 细 | 1 agent 改 5-10 个文件 | 出错范围可控，修复成本低 |
| ❌ 粗 | 1 agent 改 50 个文件 | 30+ 编译错误，修复时间反超执行时间 |

**原则 2：链式串行优于扇出并行**

当工作有隐式依赖链时（如类型定义 → API 层 → Store → View），强行并行会导致 agent 间的冲突。改为链式串行，每步验证后再下一步。

**原则 3：环境依赖前置——启动 agent 前装好所有依赖**

Agent 没有权限安装 npm 包、建目录等。必须在启动前手动准备好。

**原则 4：prompt 给出精确的文件边界 + 验证命令**

每个 agent 的 prompt 第一行写明文件范围，最后一行写明验证命令。

### 5.2 Agent 类型选择

| Agent 类型 | 适用场景 | 推荐文件数 |
|-----------|---------|:---------:|
| **单 agent 串行** | 有严格依赖链的任务 | 5-10 文件 |
| **多 agent 并行** | 互相完全独立的任务（不同模块、无共享文件） | 各 5-10 文件 |
| **混合模式** | 先串行基础层，再并行上层 | 首 agent 3-5 文件，后续各 5-10 |

### 5.3 Prompt 标准模板

每个 agent 的 prompt 应包含以下要素：

```
┌──────────────────────────────────────────┐
│ ① 文件边界：只改 XXX 目录下的文件          │
│ ② 具体任务：列出每个文件的改动             │
│ ③ 不做事项：明确排除的范围                │
│ ④ 验证命令：如何确认工作正确              │
└──────────────────────────────────────────┘
```

### 5.4 各 Phase Agent 调派方案

```
Phase A — 单 agent 顺序执行（已验证 ✅）
  └─ Agent A：CI 配置 + JWT 密钥环境变量化
     → 4-6 文件，10 min，编译验证

Phase B — 3 agent 并行（已验证 ✅）
  ├─ Agent B1：edu 核心测试（只改 test/edu/）
  ├─ Agent B2：其他模块测试 + Swagger（只改 test/ + Controller）
  └─ Agent B3：前端测试（只改 test 文件）
  → 装好 vitest 后再启动 B3 → 合并验证

Phase C — 链式串行 + 粒度控制（已验证——吸取教训后）
  批次 1（1 agent）：types/ 定义（只新建，不改现有文件）
  批次 2（1 agent）：api/*.ts 替换 any（只改 api/ 目录）
  批次 3（1 agent）：store + 5 个核心 View（最常用的页面）
  批次 4（1 agent）：剩余 View（每批 5-8 个文件）
  → 每批后运行 vue-tsc --noEmit 验证

Phase D — 链式串行，5 agent（计划）
  批次 1（Agent D1）：ScheduleConflictDetector 工具类（新建文件）
  批次 2（Agent D2）：排课 API（改 Controller + Service）
  批次 3（Agent D3）：课表多维查询 API（新建 Controller）
  批次 4（Agent D4）：前端排课管理页面（新建 View）
  批次 5（Agent D5）：侧边栏入口 + 验证

Phase E — 并行（两任务互不依赖）
  ├─ Agent E1：Redis 缓存配置（只改 pom + config + application）
  └─ Agent E2：Docker 化（新建 Dockerfile + compose）

Phase F — 单 agent 顺序
  └─ Agent F：README + 废弃代码清理
```

| API 端点路径 | R1c 文档 | 前端 API 调用、测试 mock |
| 安全路径规则 | R1a §6.3 | Swagger 配置、CI 安全检查 |
