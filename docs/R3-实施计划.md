#### C-01：前端类型系统

**实施顺序：** `types/` 基类 → API 层 → Store → View

```
步骤 1：创建 src/types/ 目录
  ├── api.ts            → Result<T>, PageResult<T>, 请求参数类型
  ├── user.ts           → User, LoginRequest, LoginResponse
  ├── course.ts         → Course, CourseSelection, Grade
  ├── admin.ts          → Notificat
ion, LeaveApplication
  ├── club.ts           → Club, ClubMember, Venue
  ├── activity.ts       → Activity, ActivityRegistration
  ├──
        → CardRecharge, LostFound
  ├── message.ts        → Conversation, Message, Announcement
  ├── todo.ts           → Todo
  └── edu.ts            → Semester, TrainingPlan, Schedule

步骤 2：api/*.ts — 替换 any 为具体类型（10 个文件）

步骤 3：store/*.ts — 替换 any（user.ts, app.ts）
```

**风险：** 类型定义需要与后端 Entity 字段一一对应，容易遗漏新字段。前端 50+ View 文件不可能一次全部改完。采用「核心页面优先，其余逐步推进」策略。

#### C-02：错误处理修复

**实施顺序：** 后端 → 前端 → 统一模式

```java
// CourseServiceImpl.java 新增方法
addSchedule(courseId, scheduleData) {
    1. 校验课程存在且未发布（已发布的不可改排课）
    2. 教室冲突检测
    3. 教师时间冲突检测

    1. 从 JSON 数组中移除指定索引
    2. 更新 course.schedule
}

updateSchedule(courseId, scheduleIndex, newData) {
    1. 替换指定索引的排课数据
    2. 重新触发冲突检测
}
```

## 4. 风险与应对

### 4.1 核心理念

> 风险不是用来「消除」的，是用来「管理」的。每个风险确认发生后，执行对应的应急预案（Plan B），而不是停下来讨论。

### 4.2 风险分级定义

| 等级 | 标识 | 说明 |
|:----:|:----:|------|
| 🔴 致命 | 项目阻塞，不可继续 | 需要立即切换方案或降级需求 |
| 🟡 严重 | 显著延迟或质量下降 | 需要主动干预，有备用方案 |
| 🟢 轻微 | 可接受范围内的影响 | 接受风险，正常推进 |
| ⚪ 提示 | 低概率但值得注意 | 保持观察，触发时再处理 |

### 4.3 风险全景矩阵

#### Phase A — CI 基础设施

| 编号 | 风险描述 | 概率 | 影响 | 等级 | 触发条件 | 应对措施 | 应急预案（Plan B） |
|:----:|---------|:----:|:----:|:----:|---------|---------|-----------------|
| A-R1 | GitHub Actions 配置语法错误，CI 不运行 | 🟡 中 | 🟡 严重 | 🟡 严重 | PR 提交后 CI 未触发或报错 | 使用 `act` 本地模拟测试 workflow | 若无 `act`，在空分支上推送测试，用 gh CLI 查看运行日志 |
| A-R2 | JWT 密钥改为环境变量后，本地开发启动失败 | 🟡 中 | 🟢 轻微 | 🟢 轻微 | `application.yml` 读取 `${JWT_SECRET}` 为空 | `application-dev.yml` 中补一个开发期默认值 | 在 `.env` 或 IDEA 环境变量中配置 `JWT_SECRET=intelligent-campus-dev-secret` |
| A-R3 | H2 测试数据库与 MySQL 方言不兼容 | 🟢 低 | 🟡 严重 | 🟢 轻微 | 特定 SQL 在 H2 下报语法错误 | 使用 `MODE=MySQL` 参数 + 避免 MySQL 特有语法 | 若 H2 不支持 `JSON_CONTAINS`，测试改用内存 MySQL（testcontainers） |

#### Phase C — 技术债务

| 编号 | 风险描述 | 概率 | 影响 | 等级 | 触发条件 | 应对措施 | 应急预案（Plan B） |
|:----:|---------|:----:|:----:|:----:|---------|---------|-----------------|
| C-R1 | 前端类型定义与实际后端 Entity 字段不一致 | 🟡 中 | 🟡 严重 | 🟡 严重 | View 编译时报 property 不存在 | 以 R1b 数据库设计为唯一来源，类型字段名与 DB 列名对齐 | 运行 `npm run dev` 观察控制台警告，逐个修正 |
| C-R2 | 空 `catch {}` 数量超过预期（可能 40+ 而非 25+） | 🟡 中 | 🟢 轻微 | 🟢 轻微 | grep 发现更多空 catch | 批量替换分两轮：先 grep 统计准确数量，再按模块分配 agent | 若超过 50 处，仅修复核心页面和 API 层 catch，非核心 View 延后到 Phase F |
| C-R3 | 添加 `@Transactional` 后出现循环依赖或事务传播问题 | 🟢 低 | 🟡 严重 | 🟢 轻微 | 启动时报 `BeanCurrentlyInCreationException` | Service 间调用使用接口而非实现类，事务注解加在 public 方法上 | 回退：仅在明确的多表写入方法上加 `@Transactional`，不全局扫描 |
| C-R4 | `vue-tsc` 类型检查通过但运行时行为异常（类型断言掩盖了真实问题） | 🟢 低 | 🟡 严重 | 🟢 轻微 | `as any` 或 `!` 非空断言导致运行时 undefined | TypeScript 配置启用 `strict: true`，减少 `as` 断言 | 对运行时错误加上 optional chaining `?.` 兜底 |
| C-R5 | 修改 GlobalExceptionHandler 后原有错误响应格式变化，前端解析失败 | 🟡 中 | 🟡 严重 | 🟡 严重 | 前端收到格式不同的错误对象 | 新增异常处理器保持与现有 `Result.error(code, msg)` 格式完全一致，不改变字段结构 | 通过 `git diff` 确认响应结构无变更 |

|:----:|---------|:----:|:----:|:----:|---------|---------|-----------------|
| D-R1 | 排课冲突检测逻辑复杂，周模式交集计算有边界 Bug | 🟡 中 | 🟡 严重 | 🟡 严重 | 单周/双周边界（第 1 周、学期最后一周）计算错误 | 冲突检测逻辑独立为 `ScheduleConflictDetector` 工具类，周模式交集写穷举测试 | 若短时间无法彻底解决，前端在冲突提示中增加「忽略冲突，强制排课」按钮（需管理员二次确认） |

| D-R5 | 排课页面课表样式与现有 ScheduleView 不一致 | 🟢 低 | 🟢 轻微 | ⚪ 提示 | 用户感觉两个课表页面"长得不一样" | 复用 ScheduleView.vue 的 CSS 样式变量和网格布局 | 统一加一个 `.schedule-grid` 共享 class |

### 4.10 文档校验 — 开发过程中保持与规格一致

**每次实施代码改动前，必须执行以下三步校验：**

```
步骤 A：定位文档
  这次改动涉及哪个模块？
  ├─ 打开 R0 找到对应功能定义
  ├─ 打开 R1c 找到对应接口定义
  ├─ 打开 R1b 确认涉及的数据库表结构
  └─ 打开 R1a 确认模块依赖和架构约束

步骤 B：对照三问
  Q1: R0 中有这个功能吗？权限定义是否匹配？
  Q2: R1c 的接口定义和我要实现的一致吗？（路径/方法/参数/响应体）
  Q3: R1b 的表结构支持我的数据需求吗？（字段名/类型/约束）

步骤 C：偏差处理
  ┌─ 实现与文档一致 → 正常开发
  ├─ 实现需要偏离文档 →
  │    先更新文档再开发（文档优先原则）
  │    改动后通知你 Review 文档变更
  └─ 文档中有但实际不需要 →
       先删除/修改文档再跳过
```

**三不原则：**
```
文档没写的功能          → 不擅自加
文档定义了的接口         → 不擅自改（格式/路径/权限）
文档中的设计决策         → 不擅自推翻
```

## 5. Agent 调派策略

### 5.1 Agent 类型选择

| Agent 类型 | 适用场景 | 在本项目中的使用 |
|-----------|---------|----------------|
| **单 agent 顺序** | 有严格依赖的任务，A → B → C | Phase A CI 配置、Phase B3 Swagger、Phase E Docker |
| **多 agent 并行** | 互相独立的任务，可同时进行 | Phase B1+B2（前后端测试并行）、Phase C1+C2（类型+错误处理并行）|
| **批量 agent** | 大量相似修改分布在不同文件中 | Phase C2 空 catch 修复（一个 agent 负责教育模块、另一个负责其他模块） |

### 5.2 各 Phase Agent 调派方案

```
Phase A（1 轮）：
  └─ Agent A：CI 配置 + JWT 密钥环境变量化
      → 4-6 文件，顺序执行，10-15 min

Phase B（3 轮并行）：
  ├─ Agent B1：后端测试（edu 核心）     │  同时进行
  ├─ Agent B2：后端测试（其他模块）     │  同时进行
  ├─ Agent B3：前端测试                 │  同时进行
  └─ Agent B4：Swagger 集成             │  独立进行（只需 1 个文件改动）
  → 完成后合并验证

Phase C（2 轮）：
  轮 1 — 并行：
    ├─ Agent C1：前端类型系统（types/ + api/ 层）
    ├─ Agent C2：错误处理修复（后端 + 前端空 catch）
    └─ Agent C3：事务注解 + 日志配置
  轮 2 — 顺序：
    └─ Agent C4：前端 View 层 any 替换（核心页面优先）
    → vue-tsc 验证通过后结束

Phase D（2 轮）：
  轮 1 — 后端：
    └─ Agent D1：冲突检测 Service + 课表多维查看 Controller
  轮 2 — 前端：
    └─ Agent D2：排课管理页面 + 课表多维查看 UI
    → 手工验证排课流程

Phase E（1 轮 — 两任务互不依赖）：
  ├─ Agent E1：Redis 缓存配置
  └─ Agent E2：Docker 化（Dockerfile + compose）

Phase F（1 轮）：
  └─ Agent F：README + 废弃代码清理
```
